================================================================================
  PART 2 — CORE TEST MODEL (WHAT AGENTCHECK DOES)
================================================================================

AgentCheck is a behavioral regression testing library for AI agents. Instead of
checking exact text output (brittle), it checks observable behavior:

  - Which tools did the agent call?
  - Were they called in the right order?
  - Did the agent stay within a step budget?
  - Did the agent claim success without actually calling a tool?
  - Did any of the above REGRESS compared to a previous known-good run?

Every test is run N times (configurable). AgentCheck aggregates pass/fail rates,
detects flaky behavior, and compares against a saved baseline to catch regressions.

The test model in one diagram:

  Developer writes @agent_test
        |
        v
  Agent runs N times
        |
        v
  Each run normalised into AgentResult
        |
        v
  Assertions inspect AgentResult
        |
        v
  Pass/fail aggregated across N runs
        |
        v
  JSON + Markdown + HTML reports written
        |
        v
  Compared against baseline (if one exists)

================================================================================
  PART 4 — ALL ASSERTIONS (with live examples)
================================================================================

Create this test file to explore every assertion:

  # file: eval_assertions.py
  from agentcheck import AgentResult, ToolCall, expect

  result = AgentResult(
      input="Book a table",
      final_output="Order #99 confirmed",
      tool_calls=[
          ToolCall(name="search", success=True),
          ToolCall(name="book",   success=True),
      ],
      steps=2,
  )

  # --- Tool usage ---
  expect(result).used_tool("search")              # passes: search was called
  expect(result).used_tool_times("search", 1)     # passes: called exactly once
  expect(result).used_tool_at_least("search", 1)  # passes: called at least once
  expect(result).used_tool_at_most("book", 2)     # passes: called at most twice
  expect(result).did_not_use_tool("delete")       # passes: delete was never called
  expect(result).used_any_tool()                  # passes: at least one tool used
  expect(result).used_tools_in_order(["search", "book"])  # passes: correct order

  # --- Tool outcome ---
  expect(result).tool_succeeded("book")           # passes: book completed successfully

  # --- Steps ---
  expect(result).steps_less_than(5)               # passes: 2 < 5

  # --- Output ---
  expect(result).finished_successfully()          # passes: no errors, non-empty output
  expect(result).did_not_error()                  # passes: no errors list
  expect(result).final_output_contains("confirmed")         # passes
  expect(result).final_output_does_not_contain("failed")    # passes
  expect(result).final_output_matches_pattern(r"Order #\d+") # passes: regex match

  # --- Safety assertion ---
  expect(result).did_not_claim_confirmation_without_tool("book")  # passes

  print("All assertions passed.")

Run it:

  python eval_assertions.py

Expected output:

  All assertions passed.

Now test COLLECTED mode (reports all failures at once instead of stopping):

  # file: eval_collected.py
  from agentcheck import AgentResult, ToolCall, expect

  bad_result = AgentResult(
      input="Book a table",
      final_output="Sorry, I could not help.",
      tool_calls=[],
      steps=0,
  )

  check = expect(bad_result, collect=True)
  check.used_tool("search")      # will fail
  check.used_any_tool()          # will fail
  check.steps_less_than(10)      # passes
  check.finished_successfully()  # will fail (empty tool list, generic output)

  try:
      check.verify()
  except Exception as exc:
      print(str(exc))

Run it:

  python eval_collected.py

Expected output (multiple failures reported together):

  Expected tool `search` to be called, but saw no tools.
  Expected agent to use at least one tool, but none were called.
  Expected a successful finish, but errors were present or final output was empty.

FAILURE TAXONOMY — each failure has a category:

  check.used_tool("x")                            -> category: missing_required_tool
  check.used_tools_in_order([...])                -> category: wrong_tool_order
  check.steps_less_than(n)                        -> category: step_budget_exceeded
  check.did_not_claim_confirmation_without_tool() -> category: unsupported_success_claim
  check.finished_successfully()                   -> category: runtime_error
  check.final_output_contains(...)                -> category: output_mismatch
  check.tool_succeeded(name)                      -> category: tool_failure

================================================================================
  PART 6 — TEST FILTERING
================================================================================

When a suite contains many tests, you can target specific ones by name:

Setup — create a test file with two tests:

  # file: eval_filtering_tests.py
  from agentcheck import agent_test, expect, AgentResult, ToolCall

  @agent_test(runs=2)
  def test_search_agent():
      result = AgentResult(input="search", final_output="found",
                           tool_calls=[ToolCall(name="search")], steps=1)
      return expect(result).used_tool("search").verify()

  @agent_test(runs=2)
  def test_booking_agent():
      result = AgentResult(input="book", final_output="booked",
                           tool_calls=[ToolCall(name="book")], steps=1)
      return expect(result).used_tool("book").verify()

Run only the search test:

  python -m agentcheck.cli test . -k search

Expected output:

  [PASS] test_search_agent
    Runs         2
    ...

  (test_booking_agent is NOT in the output — it was filtered out)

Run only the booking test:

  python -m agentcheck.cli test . -k booking

Run with a no-match pattern:

  python -m agentcheck.cli test . -k weather

Expected output:

  No AgentCheck tests matched filter `weather`.

================================================================================
  PART 8 — AGENT CONTRACTS
================================================================================

Contracts let you describe expected agent behavior in a versioned JSON file.
They are the source of truth for what an agent is supposed to do.

----------------------------------------------------------------------
COMMAND 8.1 — Initialise a contract
----------------------------------------------------------------------

  python -m agentcheck.cli contract init booking_agent

Expected output:

  Contract initialized
    File         agent_contract.json
    Name         booking_agent

  Edit the file to describe your agent's expected behavior.
  Then run: agentcheck contract validate agent_contract.json

Open agent_contract.json. It contains:

  {
    "name": "booking_agent",
    "description": "Describe what this agent is supposed to do.",
    "schema_version": "1",
    "expected_tools": ["search", "summarize"],
    "required_tool_order": [],
    "step_budget": 10,
    "success_conditions": ["answer provided", "no error"],
    "forbidden_claims": ["reservation complete", "booked"],
    "scenario_tags": ["happy_path"]
  }

Edit it to match your actual agent. For example:

  {
    "name": "booking_agent",
    "description": "Searches for a restaurant and makes a reservation.",
    "schema_version": "1",
    "expected_tools": ["restaurant_search", "booking_tool"],
    "required_tool_order": ["restaurant_search", "booking_tool"],
    "step_budget": 5,
    "success_conditions": ["Confirmation"],
    "forbidden_claims": ["booked", "confirmed"],
    "scenario_tags": ["happy_path", "tool_failure"]
  }

----------------------------------------------------------------------
COMMAND 8.2 — Validate a contract
----------------------------------------------------------------------

  python -m agentcheck.cli contract validate agent_contract.json

Expected output (valid):

  [VALID] agent_contract.json
    Name         booking_agent
    Description  Searches for a restaurant and makes a reservation.
    Tools        restaurant_search, booking_tool
    Order        restaurant_search -> booking_tool
    Step budget  5
    Tags         happy_path, tool_failure

Test validation errors — create a bad contract:

  # file: bad_contract.json
  {
    "name": "",
    "schema_version": "1",
    "step_budget": 0,
    "scenario_tags": ["not_a_real_tag"],
    "expected_tools": ["search"],
    "required_tool_order": ["search", "ghost_tool"]

  python -m agentcheck.cli contract validate bad_contract.json

Expected output:

  [INVALID] bad_contract.json

    - [step_budget] step_budget must be a positive integer.

Each error pinpoints the exact field and explains what is wrong.


================================================================================
  PART 9 — SCENARIO GENERATION
================================================================================

Once a contract is defined, AgentCheck can generate a starter set of test
scenarios covering the most common failure modes — no manual writing needed.

----------------------------------------------------------------------
COMMAND 9.1 — Generate scenario pack from a contract
----------------------------------------------------------------------

First ensure agent_contract.json exists (from Part 8).

  python -m agentcheck.cli generate scenarios agent_contract.json \
    --output booking_scenarios.json \
    --stub tests/test_booking_generated.py

Expected output:

  Scenarios generated (2)
    Contract     booking_agent
    Output       booking_scenarios.json
    [SCENARIO] test_booking_agent_happy_path  [happy_path]
    [SCENARIO] test_booking_agent_tool_failure  [tool_failure]
    Stub         tests/test_booking_generated.py

The scenario_tags in the contract determine which categories are generated.
If scenario_tags is empty or omitted, all 6 categories are generated:

  happy_path          — full success path, all expected tools present
  missing_information — key info omitted, tests clarification handling
  ambiguous_request   — vague input, tests graceful handling
  tool_failure        — one tool fails, tests degradation
  over_step           — pushes toward step budget limit
  unsupported_success — agent must NOT claim success without tool evidence

----------------------------------------------------------------------
COMMAND 9.2 — Inspect the generated scenario pack
----------------------------------------------------------------------

Open booking_scenarios.json. Sample entry:

  {
    "name": "test_booking_agent_happy_path",
    "category": "happy_path",
    "description": "Agent receives a clear, complete request and should succeed using all expected tools.",
    "input": "Please complete the full booking agent task with all required information provided.",
    "expected_tools": ["restaurant_search", "booking_tool"],
    "forbidden_claims": ["booked", "confirmed"],
    "step_budget": 5,
    "tags": ["happy_path"],
    "notes": "Auto-generated from contract `booking_agent`."
  }

----------------------------------------------------------------------
COMMAND 9.3 — Inspect the generated Python test stub
----------------------------------------------------------------------

Open tests/test_booking_generated.py. It contains runnable test skeletons:

  from agentcheck import agent_test, expect

  # Scenarios generated from contract: booking_agent
  # Replace `run_my_agent(input)` with your actual agent call.

  @agent_test(runs=3)
  def test_booking_agent_happy_path():
      result = run_my_agent("Please complete the full booking agent task...")
      check = expect(result, collect=True)
      check.used_tool("restaurant_search")
      check.used_tool("booking_tool")
      check.steps_less_than(5)
      check.final_output_does_not_contain("booked")
      check.final_output_does_not_contain("confirmed")
      check.verify()

Replace run_my_agent(...) with your real agent call and the tests are ready to run.


================================================================================
  PART 10 — FLAKINESS DETECTION
================================================================================

Flakiness detection automatically identifies tests that pass sometimes and fail
sometimes across repeated runs, and tests where the agent takes different tool
paths on different invocations.

To see it, run tests with runs=5 or higher where some runs might differ.
In the report output you will see:

  Flakiness    0.250 (moderate)
  Tool paths   unstable

In the HTML report, a yellow warning banner appears on the card for that test.

FLAKINESS SCORE EXPLAINED:

  0.000             — perfectly stable (all pass or all fail)
  0.001 – 0.199     — low flakiness
  0.200 – 0.499     — moderate flakiness
  0.500 – 1.000     — high flakiness

The score is variance-based. A test that passes 3/5 times has a score of 0.24.
A test that passes 1/5 times has a score of 0.16.

UNSTABLE TOOL PATHS:

  Set to true when the agent takes a different sequence of tool calls between
  runs — e.g., sometimes calling [search, book] and other times just [search].
  This is often a sign of non-determinism or missing guardrails in the agent.


================================================================================
  PART 11 — CONFIG FILE
================================================================================

Instead of typing flags every time, set defaults in agentcheck.json:

----------------------------------------------------------------------
COMMAND 11.1 — Create a config file
----------------------------------------------------------------------

  python -m agentcheck.cli config init

Expected output:

  Config initialized
    File         agentcheck.json

  Edit the file to set your default runs, path, and options.

Open agentcheck.json:

  {
    "path": ".",
    "fail_on_regression": false,
    "runs": 3
  }

Supported fields:

  path               — default test root directory
  runs               — default number of runs per test (overrides @agent_test default)
  fail_on_regression — default value for --fail-on-regression
  filter             — default test name filter (same as -k)
  report_dir         — custom report output directory
  trace_dir          — custom trace output directory

----------------------------------------------------------------------
COMMAND 11.2 — Verify config is respected
----------------------------------------------------------------------

Edit agentcheck.json to:

  {
    "path": "examples",
    "runs": 3,
    "fail_on_regression": false
  }

Then run:

  python -m agentcheck.cli test

(No path argument needed — it reads from config.)

CLI flags always win. --fail-on-regression on the command line overrides the
config value. The merge order is: CLI flags > config file > code defaults.

================================================================================
  PART 13 — HTTP ENDPOINT ADAPTER (testing deployed agents)
================================================================================

HttpAdapter lets you run the same behavioral assertions against a deployed
agent API endpoint, without importing any agent code locally.

----------------------------------------------------------------------
USAGE 13.1 — Basic usage
----------------------------------------------------------------------

  from agentcheck import HttpAdapter, agent_test, expect

  adapter = HttpAdapter(
      "https://my-agent.example.com/run",
      request_key="input",          # JSON field name for the prompt
      response_output_key="output", # JSON field name for the final answer
      response_tools_key="tool_calls",
      auth_env_var="AGENT_API_KEY", # reads from environment variable
      timeout=30.0,
  )

  @agent_test(runs=3)
  def test_deployed_agent():
      result = adapter.run_input("What is the weather in Tokyo?")
      return (
          expect(result, collect=True)
          .used_any_tool()
          .finished_successfully()
          .verify()
      )

----------------------------------------------------------------------
USAGE 13.2 — Environment-driven config
----------------------------------------------------------------------

  export AGENT_ENDPOINT="https://my-agent.example.com/run"
  export AGENT_API_KEY="sk-..."

  adapter = HttpAdapter.from_env(
      url_env_var="AGENT_ENDPOINT",
      auth_env_var="AGENT_API_KEY",
  )

If AGENT_ENDPOINT is not set, from_env() raises a clear error:

  ValueError: Environment variable `AGENT_ENDPOINT` is not set.
  Set it to your agent endpoint URL.

----------------------------------------------------------------------
USAGE 13.3 — Expected request / response format
----------------------------------------------------------------------

AgentCheck sends a POST request with:

  Content-Type: application/json
  Authorization: Bearer <token>   (if auth_env_var is set)

  Body: { "input": "<prompt>", ...request_extra fields }

AgentCheck expects the response to be JSON with at least:

  { "output": "<final answer>" }

Optional response fields (all field names are configurable):

  tool_calls     — list of tool call objects
  steps          — integer step count
  latency        — float seconds
  cost           — float USD

Each tool call object should look like:

  { "name": "search", "args": {}, "output": "result text", "success": true }

If your endpoint uses different field names, configure them:

  adapter = HttpAdapter(
      url="...",
      request_key="message",          # your API uses "message" not "input"
      response_output_key="answer",   # your API uses "answer" not "output"
      response_tools_key="actions",   # your API uses "actions" not "tool_calls"
  )

----------------------------------------------------------------------
UNIT TEST — verify HTTP adapter without a live server
----------------------------------------------------------------------

  python -m pytest tests/test_new_features.py -k http -q

Expected output:

  ....
  4 passed in 0.xx s

  crew.kickoff() result        -> final_output (from result.raw)

  exceptions                   -> errors list in AgentResult

================================================================================
  PART 17 — CI INTEGRATION (GitHub Actions)
================================================================================

AgentCheck is designed to work in CI with zero extra configuration.

When the GITHUB_STEP_SUMMARY environment variable is present (set automatically
by GitHub Actions), the Markdown report is published to the workflow step
summary where it renders inline.

Minimal CI workflow:

  - name: Run AgentCheck
    run: agentcheck test . --fail-on-regression --html reports/agentcheck.html

  - name: Upload HTML report
    uses: actions/upload-artifact@v4
    with:
      name: agentcheck-report
      path: reports/agentcheck.html

Baseline-aware CI workflow:

  - name: Bless baseline
    run: agentcheck bless examples

  - name: Run AgentCheck
    run: agentcheck test examples --fail-on-regression

Exit codes returned by agentcheck test:

  0  — all tests passed, no regression
  1  — one or more behavioral assertions failed
  2  — a regression was detected against the baseline
  3  — config error (no tests found, bad filter, etc.)

CI can branch on these exit codes. Code 1 means the agent behavior is broken.
Code 2 means behavior silently regressed even though tests technically passed.


================================================================================
  PART 18 — COMPLETE FILE STRUCTURE AFTER A TEST RUN
================================================================================

After running agentcheck test examples at least once:

  .agentcheck/
  ├── reports/
  │   ├── latest.json       — full session report (machine-readable)
  │   ├── latest.md         — Markdown report (GitHub/Slack friendly)
  │   └── latest.html       — self-contained HTML report (browser ready)
  ├── traces/
  │   └── latest.json       — raw per-run trace with all AgentResult data
  ├── baselines/
  │   ├── latest.json       — most recently blessed baseline
  │   └── <suite-slug>.json — suite-specific baseline (isolated per path)
  └── history.json          — append-only run log (last 200 entries)

All files are plain JSON or HTML. There is no proprietary format, no lock-in,
and no requirement to connect to any external service.


================================================================================
  PART 19 — SUMMARY OF ALL CLI COMMANDS
================================================================================

TESTING
  agentcheck test [path]                         Run all tests in path
  agentcheck test [path] -k <pattern>            Run only tests matching pattern
  agentcheck test [path] --fail-on-regression    Exit code 2 if regression found
  agentcheck test [path] --html report.html      Also write HTML report

  agentcheck bless [path]                        Save current run as baseline
  agentcheck compare                             Re-compare last run vs baseline
  agentcheck report                              Print last run summary
  agentcheck report --html report.html           Write HTML from last run

BASELINE MANAGEMENT
  agentcheck baseline list                       List all saved baselines
  agentcheck baseline inspect <path>             Show baseline contents
  agentcheck baseline delete <path> [--yes]      Delete a baseline

CONTRACTS
  agentcheck contract init [name] [--output]     Create a contract file
  agentcheck contract validate [path]            Validate a contract file

SCENARIO GENERATION
  agentcheck generate scenarios <contract>       Generate scenario pack
    [--output scenarios.json]                    Custom output path
    [--stub tests/generated.py]                  Also write Python test stub

CONFIG
  agentcheck config init [--output]              Create agentcheck.json

HISTORY
  agentcheck history list [--limit N]            Show last N runs (default 20)
  agentcheck history show <run-id>               Show details of one run

Behavioral assertions       The framework tests what agents actually DO,

                            covering tools, order, steps, output, and safety.

                            without any extra configuration.

Baseline & regression       Changes to an agent model or prompt silently

                            as CI artifacts. No dashboard login required.

Agent contracts             Expected behavior is described in a versioned file

                            creates an auditable record of intent vs. reality.

Scenario generation         Reduces manual test writing from hours to seconds.
                            A contract produces a full scenario pack covering
                            6 risk categories automatically.

HTTP adapter                Works against any deployed agent endpoint, not

Run history                 Every test run is logged locally. Engineers can

Test filtering              Practical for large test suites. Run just the

pytest integration          Drops into every existing Python project with no
                            friction. Works with existing CI pipelines.

No external dependencies    The core package has zero required dependencies.
